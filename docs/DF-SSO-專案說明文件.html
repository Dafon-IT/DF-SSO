Part I

系統與技術設計

這個系統解決什麼問題、長什麼樣子、登入怎麼跑、做了哪些工程決策。

1. 專案摘要

DF-SSO 是大豐環保的「企業統一登入（Single Sign-On）」中央系統。 它解決一個很實際的問題：公司內部系統愈來愈多（CRM、ERP、各式後台），如果每套系統各做一份登入、各管一份帳號密碼，使用者要記很多組密碼、IT 要重複維護很多套登入機制，安全性也難統一控管。 DF-SSO 把「登入」這件事集中到一個地方 —— 所有系統都導來這裡，使用者用同一組 Microsoft（Azure AD）公司帳號登入一次，就能通行所有已接入的系統，不必重複登入、不必各記密碼。

它替公司做到什麼

  • 一次登入、到處通用：員工登入一個系統後，切換到其他已接入系統免再登入。
  • 帳號集中、共用公司身分：直接沿用 Microsoft 公司帳號，員工到職 / 離職由 AD 統一控管，不必在每套系統各開各關帳號。
  • 安全集中控管：登入紀錄、授權範圍、防暴力登入（Rate Limit）等都在中央一處管理。
  • 新系統接入快：要接 SSO 的系統只需向中央申請一組憑證（app_id / app_secret），照整合文件接四個端點即可，不用自己重做登入。

系統由哪些部分組成

  • 登入中央（後端）：負責跟 Microsoft 驗證、發憑證、管理登入狀態，是各系統之間溝通的核心。
  • 管理後台（網頁）：管理員登入操作的地方，可新增 / 停用要接入的系統、查誰在什麼時候登入、管理後台管理員、調整安全參數。本手冊大部分操作都在這裡進行。
  • 身分來源：Microsoft Azure AD（公司帳號）；登入時另會向 ERP 取員工資料（員編、部門等）。

誰在用

所有接入 SSO 的內部系統，以及它們背後的使用者（公司員工）。日常會用到這份手冊的人，是負責維運 SSO 的 IT 人員 / 接手者。

目前狀態

  • 正式（Prod）與測試（Test）兩套環境各自獨立運作中，資料庫與憑證完全隔離、不共用。
  • 部署在 Coolify（Docker Compose）平台上。
  • 日常維運工作集中在兩類：① 幫新系統接入（發憑證）、② 處理「登不進去」的問題（查登入紀錄定位）。這兩類的操作步驟見第 9 節 Playbook。
接手者最該先知道的三件事： ① Azure AD 的 client secret 會過期，過期將導致全公司登入中斷 —— 務必先確認到期日與負責續期的人； ② Prod 的資料庫操作不可逆，誤跑會資料全毀（見第 10 節紅線）； ③ 其餘風險與待補的營運資訊見「已知風險與待辦」一節。

2. 系統架構

DF-SSO 是一個獨立的 OAuth2 Authorization Server，前後端分離，搭配 PostgreSQL（持久資料）與 Redis（登入狀態），對外接 Microsoft Azure AD 與 ERP API。所有接入應用（Client App）只跟「中央」對話，不直接碰 Microsoft。

技術棧

後端

  • Node.js · Express
  • OAuth2 Authorization Code Flow
  • JWT（HS256）+ Redis Session
  • node-pg-migrate（雙目錄）

前端

  • Next.js（App Router）
  • 單頁 Tab Dashboard
  • cookie-based（credentials: include）

資料 / 狀態

  • PostgreSQL — 白名單 / log / 管理員
  • Redis — session + 一次性 auth code

外部整合

  • Microsoft Azure AD（MSAL）
  • ERP API（員工資料）

部署 / 維運

  • Coolify · Docker Compose
  • Prod / Test 雙獨立 stack
  • Helmet / CORS 動態白名單 / Rate Limit

安全

  • timingSafeEqual 常數時間比對
  • Redis Lua 原子 GET+DEL
  • HMAC-SHA256 back-channel
環境隔離：Prod 與 Test 是兩套獨立的 Coolify stack，PostgreSQL / Redis volume 完全隔離（sso-prod-* vs sso-test-*），各自發各自的 app_id / app_secret。

3. SSO 完整 OAuth2 登入流程（§7.1.1）

完整 Authorization Code Flow，含 cookie / client_id / redirect_uri / state / client_secret / code 各驗證分支與失敗導向。每個 🛑 都是一個被擋下的攻擊面。

關鍵性質：① 中央 Redis 是登入狀態的唯一事實來源（JWT 簽名有效 ≠ 還在登入）；② 跨 App 免登入靠中央 session 跳過 Microsoft 那一段；③ 登出走中央 HMAC-SHA256 back-channel 通知所有接入 App；登出與 silent re-auth 兩種模式取捨見 docs/Design.md。

4. 技術亮點與工程決策

這節說明系統在安全、正確性、可維運性上做的關鍵決策——每一條都是「遇到什麼問題 → 怎麼解 → 帶來什麼價值」。

4.1 安全原語

機制解決的問題怎麼做
常數時間比對 Timing attack — 用回應時間差猜出 secret / state client_secret、OAuth state 一律 crypto.timingSafeEqual（先比長度再比內容）
一次性 auth code 原子消耗 Race condition — 同一個 code 被重放 / 重複兌換 Redis Lua script GET+DEL 單一原子操作，code 一經兌換即刪，TTL 60s
Back-channel logout 簽章 偽造登出通知 / replay 舊通知 payload 帶 HMAC-SHA256(app_secret, user_id:timestamp)，Client 必須驗簽章 + 30 秒 timestamp 窗
Session fixation 防護 攻擊者預先固定 session id，登入後接管 登入成功一律 req.session.regenerate() 換新 session
CSRF（state 驗證） 偽造授權回呼 Microsoft callback 的 state 以常數時間比對，不符即 invalid_state
Redirect URI 收斂 Open redirect / path 注入 只存 / 比對 new URL(uri).origin，剝除 path/query/fragment；每 App 上限 10 筆、限 http/https

4.2 單一事實來源（Single Source of Truth）

  • 登入狀態的唯一事實來源是中央 Redis session，不是 JWT。JWT 簽名有效 ≠ 還在登入——所以「停用」「登出」之所以即時有效，是因為中央 session 被清掉/擋掉。
  • 白名單 sso_allowed_list 同時驅動五件事：OAuth2 授權、Code exchange、CORS 允許來源（快取 60 秒）、Back-channel logout、Logout redirect 驗證。一張表治理整個信任邊界，改一處五處同步。

4.3 動態 Rate Limit（可運維、零部署調整）

四層限流（全域 / Auth / Session / Exchange）不寫死在程式碼，由 sso_setting 表動態載入，管理員在後台即時調整、PUT 成功後 reload() 重建 limiter。若 DB 不可用則 fallback 到內建預設值——兼顧可調整與可用性。

範圍預設防護目標
全域500 / 15minDoS
Auth（login / authorize）30 / 15min暴力登入
Session（/me / logout）100 / 15min高頻 server-to-server
Exchange20 / 1minauth code 猜測

4.4 可維運性與環境隔離

  • 環境變數集中治理：所有 process.env 只在 config/index.js 讀，啟動時驗證必填變數——避免散落各處、缺值才在執行時爆炸。
  • 雙 migration 目錄：migrations/dev/（歷史完整）與 migrations/prod/（乾淨 baseline），由 MIGRATIONS_DIR 切換；down 絕不刪管理後台自身白名單，避免毀掉後台自己的存取權。
  • Prod / Test 完全隔離：兩套 Coolify stack，Postgres / Redis volume 不共用，憑證各自獨立。
  • 前後端分離 + CORS 動態白名單：CORS 允許來源由白名單即時推導（快取 60 秒），新增/停用應用不需改 server 設定或重部署。
一句話總結技術定位：DF-SSO 是一個把「身分驗證、session 治理、跨 App 自動登入、安全限流」收斂到單一中央、並以最小信任邊界 + 單一事實來源為設計主軸的 OAuth2 Authorization Server。

6. 第一次接手：登入管理後台

  1. 開瀏覽器到上面的管理後台網址（正式環境用 Prod 那欄）。
  2. 點登入 → 會導到 Microsoft 登入頁 → 用你的公司帳號登入。
  3. 登入後若看到 Dashboard，代表你已經是管理員了。
  4. 若登入後被踢回登入頁 / 顯示沒有權限 → 代表你的 email 還不在管理員白名單。請現任管理員到「管理員」分頁把你的 email 加進去（見 Playbook D），你再重新登入一次即可。
管理員身分驗證是三層：Microsoft 帳號有效 → 中央 session 存在 → 你的 email 在管理員白名單且為啟用狀態。三者缺一就進不來。

7. 後台四個分頁導覽

8. 核心觀念（處理需求前一定要懂的 5 件事）

  1. app_id / app_secret 是一個系統接入 SSO 的鑰匙。由後台「新增應用」時自動產生，你不用自己想。app_secret 只有在後台按「顯示」時看得到完整值，列表平常只存末四碼。
  2. redirect_uris 存的是「origin」不是完整網址。例如填 https://crm.df-recycle.com.tw，不要填 .../login/callback。一個系統最多 10 筆（給 dev/test/prod 不同網域用），只接受 http / https。
  3. 「停用」可逆，「刪除」是軟刪除。停用＝該系統暫時無法登入但資料留著；刪除＝資料庫標記，非真的清掉。兩者都不會動到使用者的 Microsoft 帳號。
  4. 登入狀態的唯一事實來源是中央的 Redis session，不是 token。所以「登出」「停用」之所以有效，是因為中央 session 被清掉 / 擋掉。
  5. 改白名單會同時影響五件事：OAuth 授權、換 token、CORS 允許來源（快取 60 秒）、登出廣播、登出 redirect 驗證。所以停用一個 App 後，最多 60 秒內它的跨網域請求才會真的被擋。

9. 常見使用者需求 Playbook（照著點）

以下每個情境都假設你已登入正式環境的管理後台。測試環境同理，只是換到 Test 後台操作。

Playbook A：「幫我把 ○○ 系統接進 SSO」

這是最常見的需求。你的工作是發憑證，對方工程師負責在他的系統裡接。

你在後台要做的：

  1. 進「白名單管理」分頁。
  2. 右上角點「＋ 新增應用」。
  3. 填表單：
    • 網域 *（必填）：對方系統正式網址 origin，例如 https://crm.df-recycle.com.tw。必須是合法 http/https，否則被擋。
    • 系統名稱：給人看的名字，例如「CRM 系統」。
    • 說明：一句話描述用途（可空白）。
    • Redirect URIs：每行一個 origin，dev/test/prod 各一行，最多 10 筆。例如： 
      http://localhost:3100
https://crm-test.apps.zerozero.tw
https://crm.df-recycle.com.tw
    • Frontend URL / Backend Docs URL：選填，方便日後快速點開該系統首頁 / API 文件。
  4. 按「建立應用」。系統會自動產生 app_id 和 app_secret。
  5. 在該應用那一列，點 App Secret 欄位的眼睛圖示「顯示」→ 出現完整 app_secret。
  6. 點旁邊的複製圖示，把 app_id 和 app_secret 複製下來。
  7. 安全地把這兩個值交給對方工程師，並請他讀 INTEGRATION.md 自行接入。

交付給對方的東西：

  • app_id、app_secret
  • SSO Backend 網址（依環境給 Prod 或 Test）
  • 一句話：「請照 INTEGRATION.md 接 callback / me / logout / back-channel-logout 四個端點」
⚠️ 安全提醒：app_secret 等同密碼。不要用一般聊天 / email 明文傳。用公司核可的密碼傳遞方式。🔲 待補（口述）：公司目前用什麼方式傳 secret？
💡 後續多了新網域（例如多開一個測試站）→ 不用重發憑證，直接「編輯」這個應用，在 Redirect URIs 多加一行即可（上限 10）。

Playbook B：「某某員工登不進去 / 某系統登入有問題」

先別急著改設定，用登入紀錄定位問題。

  1. 進「登入紀錄」分頁。
  2. 在「Email」欄輸入該員工公司 email → 按「搜尋」。
  3. 可用「狀態」「開始/結束日期」縮小範圍。
  4. 看最近一筆的「狀態」欄：
狀態意思通常怎麼處理
成功 successSSO 這端登入成功問題多半在對方系統那端（token 驗證 / cookie / redirect），請對方工程師查
失敗 failedMicrosoft 認證或流程失敗確認該員工 Microsoft 帳號正常、密碼沒過期；多人同時發生才可能是系統問題
ERP 未找到帳號登入了，但在 ERP 對不到員工資料該員工 ERP 員編 / 資料有問題，需找 ERP 端確認 🔲 待補：ERP 由誰維護？
  • 若完全查不到該員工任何紀錄 → 請求根本沒到 SSO，問題在對方系統「沒登入時要導來 SSO」那段沒接好，或網域沒在 Redirect URIs 裡。
  • 若整個系統所有人都登不進去 → 先確認該應用在「白名單管理」是不是被「停用」了；再看第 11 節故障排除。

Playbook C：「○○ 系統要下線 / 暫停 / 憑證外洩」

情況一：暫時停用（之後可能再開）

  1. 進「白名單管理」，找到該應用。
  2. 點該列「狀態」欄的綠色「啟用」標籤 → 變灰色「停用」。
  3. 完成。該系統最多 60 秒後無法再登入（CORS 快取到期）。要恢復就再點一次。

情況二：app_secret 外洩，要換新的

  1. 進「白名單管理」，找到該應用。
  2. 先點 App Secret 的眼睛圖示「顯示」（「重新產生」按鈕只有在顯示憑證時才出現）。
  3. 點「重新產生」→ 確認框按確定。
  4. 舊 secret 立刻失效，畫面跳出新的 secret。
  5. 立即複製新 secret，交給對方工程師更新環境變數。
⚠️ 在對方更新完成前，他的系統會換不到 token（登入會壞），所以這動作要跟對方講好時間再做。

情況三：永久下線

  1. 確認真的不要了（軟刪除可救，但別當常態操作）。
  2. 進「白名單管理」，找到該應用，點「刪除」→ 確認。

Playbook D：「新增 / 移除一位後台管理員」

「管理員」＝能登入這個後台的人，不是一般系統使用者。

新增：

  1. 進「管理員」分頁 → 右上角「新增管理員」。
  2. 只需填對方公司 email（會自動轉小寫）→ 按「新增」。
  3. 對方姓名 / Azure 資訊會在他首次登入後自動補上（在那之前狀態顯示「未登入」）。
  4. 請對方自己去管理後台用 Microsoft 帳號登入一次即可生效。

停用 / 刪除：

  • 停用（保留資料、暫時不給進）：在該管理員那列點「啟用/停用」標籤切換。
  • 刪除：點該列「刪除」→ 確認。
⚠️ 系統禁止你刪除自己（會跳「無法刪除自己」），這是防呆，正常現象。要移除自己請找另一位管理員操作。

Playbook E：「登入一直被擋 / too many requests / 要調次數上限」

SSO 有 Rate Limit（每個 IP 在一段時間內的請求上限），防暴力登入與 DoS。可在後台動態調整、改完立即生效。

  1. 進「設定」分頁 → 找「速率限制（Rate Limit）」區。
  2. 會看到四組可調的限制：
設定 key預設管什麼
rate_limit.global500 / 15 分鐘全域總量，防 DoS
rate_limit.auth30 / 15 分鐘登入 / 授權，防暴力登入
rate_limit.session100 / 15 分鐘系統間高頻呼叫（/me、登出）
rate_limit.exchange20 / 1 分鐘換 token，防猜測
  1. 改「視窗長度（windowMs，毫秒）」或「最大請求數（max）」→ 卡片右上出現「未儲存」→ 按「儲存」。
  2. 儲存後立即生效，該限制計數會歸零重來。
⚠️ 調大上限會降低安全防護，請有理由再調（例如某系統是 server-to-server 高頻呼叫才需調 session）。不確定就先別動，問前手。若資料庫暫時連不上，系統會自動 fallback 回上表預設值。

10. 維運操作（需要進伺服器 / 跑指令的）

這些不是日常點按操作，是工程師偶爾才會做的。指令細節見 CLAUDE.md。

我要做什麼怎麼做
看後端 log到 Coolify 對應 stack 看 container log
重啟服務Coolify 上 redeploy / restart 對應 stack
改環境變數Coolify 該 stack 的環境變數設定，改完要 redeploy
套用資料庫變更（migration）Prod 用 npm run migrate:up:prod；Dev/Test 用 npm run migrate:up:dev
本機跑起來開發backend/ 跑 npm run dev；frontend/ 跑 npm run dev
⚠️ Migration 紅線（務必遵守）：
  • 不要手動在 Prod 跑 migrate:down — Prod baseline 的 down 會 drop 掉三張表，資料全毀。
  • Schema 改動要 dev 和 prod 兩個目錄都加。
  • migration 的 down 絕不刪除管理後台自己網域的白名單資料（否則會把後台自己的存取權毀掉）。

11. 故障排除 Runbook

症狀可能原因先檢查什麼
所有人、所有系統都登不進去中央 Redis / Postgres 掛了，或服務沒起來Coolify 看 backend container 是否運行、Redis/Postgres 是否健康；看 backend log
單一系統全部人登不進去該 App 被停用 / 憑證被重設沒同步 / Redirect URI 沒包含其網域白名單管理確認該 App「啟用」中、Redirect URIs 含其 origin
單一使用者登不進去個人帳號 / ERP 資料問題走 Playbook B 查登入紀錄
跨系統免登入失效（每個系統都要重登）中央 session 沒建立 / cookie 被擋確認 session 正常、瀏覽器第三方 cookie 設定
出現 CORS 錯誤該系統 origin 不在白名單，或剛改完還在 60 秒快取內確認 Redirect URIs / 網域有填對，等 60 秒再試
改了 Rate Limit 沒生效—確認有按「儲存」、看到提示；理論上即時生效
登出後又自動被登回去silent re-auth 行為，屬設計取捨見 Design.md 的「登入頁兩種模式」與「Single Logout 加強模式」
大部分「登入成功但系統還是怪怪的」問題，根因在對方系統那端的接入，不在 SSO 中央。用登入紀錄確認「SSO 這端是 success」就可以把球丟回去給對方工程師查。
